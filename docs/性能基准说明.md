# 性能基准说明

## 目的

这份文档记录 `paper-format-normalizer` 当前的基准测试入口、观测指标，以及一份真实论文样例的实测结果，方便后续持续比较优化前后差异。

## 基准命令

```powershell
$env:PYTHONPATH='C:\Users\admin\Desktop\JJBand\src'
python -c "from paper_format_normalizer.cli import main; main()" benchmark --input <文档路径> --rules C:\Users\admin\Desktop\JJBand\规则化文件夹 --output-dir C:\Users\admin\Desktop\JJBand\输出文档 --repeat 3
```

说明：

- `--input`：待测文档，支持 `.docx`，`.doc` 会先自动转成 `.docx`
- `--rules`：规则目录
- `--output-dir`：基准结果和规范化产物输出目录
- `--repeat`：重复执行次数，输出平均值

## 输出内容

基准命令会：

- 正常生成规范化文档
- 正常生成修改报告
- 正常生成红字标注版
- 额外生成一份 `benchmark_summary.csv`

当前基准汇总字段包括：

- `prepare_input_seconds`
- `load_rules_seconds`
- `parse_docx_seconds`
- `classify_document_seconds`
- `load_document_seconds`
- `apply_rules_seconds`
- `save_normalized_docx_seconds`
- `write_report_seconds`
- `build_annotation_plan_seconds`
- `write_annotated_docx_seconds`
- `total_seconds`

## 结果解读

当前这份真实样例里，主要结论是：

1. `.doc -> .docx` 转换仍然是最大的单项成本。
   `prepare_input_seconds` 明显高于后续原生 `.docx` 规范化链路。

2. 原生规范化主链路已经比较轻。
   解析、分类、规则应用合计大约在 `1.7s` 左右。

3. 红字标注版写出已经明显优化。
   通过复用内存中的规范化文档副本，`write_annotated_docx_seconds` 已降到约 `0.52s`。

## 当前优化结论

截至这份文档，对性能已经完成了这些低风险优化：

1. 统一运行时索引缓存。
   正文、页眉、页脚、表格映射统一构建并复用。

2. 去掉主路径的 location 反解析。
   `normalize_document` 不再依赖 header/footer location 的字符串回拆。

3. 前移红字标注计划构建。
   先构建 `_AnnotationPlan`，再执行标注写出。

4. 复用内存中的规范化文档生成红字版。
   避免为红字标注再次打开规范化 `.docx`。

## 下一步建议

如果继续做性能相关工作，优先级建议是：

1. 继续观测 `.doc` 转换链路。
   这是当前最主要的单项耗时来源。

2. 增加多文档 benchmark。
   对不同学校模板、不同页数、不同表格复杂度的论文建立对比样本。

3. 将 benchmark 结果长期化。
   后续可以在每轮性能优化后更新这份文档，形成可持续对照。
